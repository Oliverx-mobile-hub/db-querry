# 功能规格：数据库查询工具

**功能分支**：`001-database-query-tool`

**创建日期**：2026-07-15

**状态**：草稿

**输入**：用户描述：“这是一个数据库查询工具，用户可以添加一个 DB url，系统会连接到数据库，获取数据库 metadata，然后将数据库中的 table 和 view 信息展示出来。用户可以自己输入 SQL 查询，也可以通过自然语言生成 SQL 查询。数据库连接字符串和数据库 metadata 都会存储到 SQLite 数据库中。系统可以根据 PostgreSQL 的功能查询系统表和视图信息，然后用 LLM 将这些信息转换成 JSON 格式并存储到 SQLite 数据库中，以后复用。当用户使用 LLM 生成 SQL 查询时，可以把系统中的表和视图信息作为 context 传递给 LLM。任何输入的 SQL 语句都需要经过服务端解析，确保语法正确，并且仅包含 SELECT 语句；如果语法不正确，需要给出错误消息。如果不含 LIMIT 子句，则默认添加 LIMIT 1000 子句。输出格式是 JSON，前端将其组织成表格并显示出来。”

## 用户场景与测试 *(必填)*

### 用户故事 1：添加数据库连接并采集 metadata（优先级：P1）

用户输入一个数据库连接 URL，系统通过后端连接目标数据库，验证连接可用性，读取数据库中的表和视图信息，并将结构化 metadata 保存到本地 SQLite 中，供后续浏览、SQL 生成和查询执行复用。

**为什么是这个优先级**：没有可用连接和 metadata，后续的 schema 浏览、自然语言生成 SQL 和 SQL 查询都无法成立。

**独立测试**：使用一个可访问的 PostgreSQL 连接 URL 创建连接；系统应完成连接测试、采集 table/view metadata、保存 metadata，并返回不包含明文连接字符串的连接信息和采集状态。

**验收场景**：

1. **假设** 用户提供有效的 PostgreSQL 连接 URL，**当** 用户提交添加连接，**那么** 系统保存连接记录、验证连接成功、采集 table/view metadata，并返回连接 ID、显示名称和 metadata 状态。
2. **假设** 用户提供格式错误、凭据无效或无法访问的 DB URL，**当** 用户提交添加连接，**那么** 系统返回用户可理解的错误消息，并且不得向前端泄露密码、完整连接字符串、堆栈或底层连接细节。
3. **假设** metadata 采集过程中部分对象无法解析，**当** 采集流程结束，**那么** 系统保存可用 metadata，并记录脱敏后的采集警告。

---

### 用户故事 2：浏览数据库中的表和视图（优先级：P1）

用户选择已保存的数据库连接后，可以查看数据库中的表、视图、字段、字段类型、可空性、主键信息和对象说明，用于理解当前数据库可查询的数据范围。

**为什么是这个优先级**：metadata 浏览是用户手写 SQL 和自然语言生成 SQL 的基础，也能让用户确认系统对数据库结构的理解是否正确。

**独立测试**：选择一个已完成 metadata 采集的连接；前端应能加载表和视图列表，并展示每个对象的字段详情。

**验收场景**：

1. **假设** 连接已存在且 metadata 已采集，**当** 用户打开该连接，**那么** 系统展示 table/view 列表、字段详情和 metadata 更新时间。
2. **假设** 连接存在但 metadata 为空或采集失败，**当** 用户进入 metadata 页面，**那么** 前端展示空状态或错误状态，并提供重新采集 metadata 的操作。
3. **假设** 用户触发重新采集 metadata，**当** 后端完成采集，**那么** SQLite 中保存的 metadata 被更新，并展示新的状态和更新时间。

---

### 用户故事 3：执行安全的只读 SQL 并查看 JSON 结果（优先级：P1）

用户可以输入 SQL 查询。后端必须解析并校验 SQL，只允许只读 SELECT 查询；语法错误、禁止语句和多语句输入必须被拒绝。合法查询执行后，后端返回 JSON 结果，前端将其组织成表格展示。

**为什么是这个优先级**：安全查询执行是工具的核心能力，也是项目宪章中风险最高、必须测试覆盖的关键路径。

**独立测试**：分别提交合法 SELECT、缺少 LIMIT 的 SELECT、语法错误 SQL、包含 UPDATE/DELETE/DROP 的 SQL、多语句 SQL；验证校验结果、默认 LIMIT 行为、错误脱敏和 JSON 输出。

**验收场景**：

1. **假设** 用户输入合法 `SELECT` 查询且包含 `LIMIT`，**当** 用户执行 SQL，**那么** 后端执行查询并返回列信息、行数据、行数和执行耗时。
2. **假设** 用户输入合法 `SELECT` 查询但未包含 `LIMIT`，**当** 用户执行 SQL，**那么** 后端执行前默认限制最多 1000 行，并在响应中说明已应用默认限制。
3. **假设** 用户输入语法错误 SQL，**当** 用户执行 SQL，**那么** 后端拒绝执行并返回语法错误消息。
4. **假设** 用户输入修改性、破坏性、权限变更、过程调用或多语句 SQL，**当** 用户执行 SQL，**那么** 后端拒绝执行并返回安全校验错误。
5. **假设** 查询结果为空，**当** 后端返回执行结果，**那么** 响应包含列信息、空行数组和空结果标记，前端展示空结果状态。

---

### 用户故事 4：通过自然语言生成 SQL（优先级：P2）

用户选择一个数据库连接并输入自然语言问题。后端将该连接已保存的 table/view metadata 作为上下文传递给 LLM，生成只读 SQL 草稿。生成结果必须先返回给用户预览，并经过与手写 SQL 相同的安全校验后才能执行。

**为什么是这个优先级**：自然语言生成 SQL 是重要增强能力，但必须建立在 metadata 可用和 SQL 安全校验可靠的基础上。

**独立测试**：对一个已有 metadata 的连接提交自然语言问题；系统应返回 SQL 草稿、说明和校验状态，且不得自动执行生成的 SQL。

**验收场景**：

1. **假设** 连接 metadata 可用，**当** 用户提交自然语言问题，**那么** 后端生成 SQL 草稿，并返回 SQL、简短说明、涉及的表或视图和校验状态。
2. **假设** LLM 生成的 SQL 不满足只读安全规则，**当** 后端校验生成结果，**那么** 系统不得将其标记为可执行，并提示用户修改问题或 SQL。
3. **假设** 连接 metadata 缺失、过期或不可用，**当** 用户请求生成 SQL，**那么** 系统提示用户先采集或刷新 metadata。

---

### 边界情况

- DB URL 格式正确但凭据无效、网络超时、数据库拒绝连接或权限不足。
- 数据库包含多个 schema、同名表、大小写敏感标识符、保留字字段名或需要双引号的对象名。
- 表或视图数量很大，metadata 采集和 LLM 上下文需要有可控大小和明确截断策略。
- metadata 中存在复杂类型、数组类型、JSON 类型、自定义类型、缺失注释或 LLM 难以稳定转换的字段说明。
- SQL 中包含注释、CTE、子查询、窗口函数、函数调用、字符串中的禁止关键字或大小写混合关键字。
- 用户输入多条语句、事务语句、临时表、函数执行、`SELECT ... INTO` 或可能通过函数造成副作用的查询。
- 查询执行超时、结果集过大、列名重复、数据库返回二进制或非 JSON 友好类型。
- LLM 服务不可用、超时、返回非 SQL 文本或返回与 metadata 不匹配的 SQL。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统必须允许用户通过后端提交 DB URL 来创建数据库连接。
- **FR-002**：系统必须将数据库连接记录保存到 SQLite，并且不得向前端返回原始凭据或完整 DB URL。
- **FR-003**：系统必须在后端验证数据库连接可用后，才将连接标记为可用。
- **FR-004**：系统必须采集 PostgreSQL 表和视图 metadata，至少包括 schema 名、对象名、对象类型、字段名、字段类型、可空性、主键信息和说明信息。
- **FR-005**：系统必须将采集到的 metadata 转换为结构化 JSON，并保存到 SQLite 以便后续复用。
- **FR-006**：系统必须允许用户刷新已有连接的 metadata。
- **FR-007**：系统必须允许用户查看已保存连接列表，但不得暴露数据库凭据。
- **FR-008**：系统必须允许用户查看某个连接下的表、视图、字段和 metadata 状态。
- **FR-009**：系统必须允许用户通过后端提交 SQL 文本进行校验和执行。
- **FR-010**：系统必须在执行前解析并校验每一条用户提交的 SQL。
- **FR-011**：系统必须拒绝语法错误 SQL、多语句 SQL、非 SELECT SQL、修改性 SQL、破坏性 SQL、权限变更 SQL、存储过程调用和执行语句。
- **FR-012**：系统必须对 LLM 生成的 SQL 应用与手写 SQL 完全相同的校验规则。
- **FR-013**：系统必须在合法 SELECT 查询没有 LIMIT 子句时，默认限制最多返回 1000 行。
- **FR-014**：系统必须限制查询执行时间和最大结果规模。
- **FR-015**：系统必须以 JSON 格式返回查询结果，包含列信息、行数据、行数、执行耗时和 LIMIT 信息。
- **FR-016**：系统必须对 API 错误进行规范化和脱敏，错误信息应便于用户理解，但不得泄露密钥、堆栈或原始连接详情。
- **FR-017**：系统必须允许用户针对选定连接提交自然语言问题，并返回生成的 SQL 草稿。
- **FR-018**：系统必须在调用 LLM 生成 SQL 时使用该连接已保存的 table/view metadata 作为上下文。
- **FR-019**：系统必须将生成的 SQL 作为预览结果返回，不得自动执行。
- **FR-020**：系统返回给前端的 JSON 字段名必须使用 camelCase。
- **FR-021**：系统必须支持无需认证、登录、角色或会话管理即可使用。
- **FR-022**：数据库访问、SQL 校验、SQL 生成编排和查询执行必须由 Go 后端完成。
- **FR-023**：前端必须作为 Vue 3 + Element Plus + TypeScript 薄客户端，负责连接表单、metadata 浏览、SQL 编辑、SQL 预览、结果表格、加载状态和错误展示。

### API 契约范围

本规格只定义 API 的能力边界和行为要求，详细请求/响应字段、错误码、DTO 类型和 OpenAPI 描述应在后续 `contracts/` 中定义。

本功能至少需要覆盖以下 API 能力：

- 创建并测试数据库连接。
- 查询已保存连接列表。
- 查询指定连接的 metadata。
- 刷新指定连接的 metadata。
- 校验 SQL，但不执行。
- 执行通过校验的 SQL。
- 根据自然语言和 metadata 生成 SQL 草稿。

所有 API 响应必须使用一致的 JSON 结构，且字段名使用 camelCase。具体响应 envelope 和错误结构在 plan 阶段固定。

### 数据交互模型范围

详细的数据交互模型不在本规格中展开，应在后续 `plan.md`、`data-model.md` 和 `contracts/` 中完成，包括：

- 前端、Go 后端、SQLite、目标 PostgreSQL、LLM 之间的数据流。
- SQLite 表结构和字段约束。
- Go 后端请求/响应结构体。
- TypeScript API 类型。
- metadata JSON schema。
- 查询结果 JSON schema。
- SQL 校验结果模型。
- 错误码、错误结构和脱敏策略。

### 关键实体

- **数据库连接**：表示一个已保存的目标数据库连接，包含连接 ID、可展示名称、数据库类型、安全展示信息、创建时间、更新时间和 metadata 状态；不得向前端暴露原始连接字符串或密码。
- **Metadata 快照**：表示某个连接在某次采集后得到的结构化数据库结构信息，包含连接 ID、状态、对象数量、更新时间和采集警告。
- **数据库对象 Metadata**：表示一个表或视图，包含 schema、对象名、对象类型、说明和字段集合。
- **字段 Metadata**：表示表或视图中的一个字段，包含字段名、字段类型、可空性、顺序、主键标记和说明。
- **SQL 校验结果**：表示 SQL 解析和策略校验后的结果，包含是否有效、错误原因、是否应用默认 LIMIT 和可执行状态。
- **查询执行结果**：表示一次只读查询的 JSON 输出，包含列信息、行数据、行数、执行耗时、LIMIT 信息和空结果标记。
- **生成 SQL 草稿**：表示一次自然语言生成 SQL 的结果，包含用户问题、SQL 草稿、简短说明、引用对象和校验状态。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：用户可以添加一个有效 PostgreSQL 连接，并在不手动编辑配置文件的情况下看到 table/view metadata。
- **SC-002**：100% 的用户提交 SQL 和 LLM 生成 SQL 都必须在后端执行前经过 SQL 校验。
- **SC-003**：系统能在执行前拒绝 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`、`CREATE`、`REPLACE`、`GRANT`、`REVOKE`、`EXEC`、存储过程调用和多语句输入。
- **SC-004**：合法 SELECT 查询没有 LIMIT 时，默认最多返回 1000 行。
- **SC-005**：查询响应是 JSON 可序列化数据，前端可以直接组织成表格展示，无需再调用数据库。
- **SC-006**：LLM 生成的 SQL 不会被自动执行，并且如果不满足只读规则会被拒绝。
- **SC-007**：API 响应不得暴露原始 DB URL、密码、模型密钥、堆栈信息或后端专用配置。
- **SC-008**：metadata 会从 SQLite 复用，除非用户刷新 metadata 或当前 metadata 被标记为不可用。

## 假设

- v1 首先支持 PostgreSQL 或 PostgreSQL 兼容数据库，因为需求明确提到通过 PostgreSQL 系统表和视图获取 metadata。
- SQLite 是本地应用数据库，用于保存连接记录、metadata 快照，以及后续可能添加的 SQL 草稿或查询历史。
- DB URL 的保护方式属于实现计划阶段的设计内容，可以是加密存储、系统密钥管理或后端私有配置引用，但不得暴露给前端。
- 用户描述中的“经过 sqlserver 解析”理解为“在服务端进行 SQL 解析和校验”，不是指必须支持 Microsoft SQL Server 方言。
- authentication 明确不在当前范围内，本功能不引入登录、用户、角色或会话。
- LLM 输出必须视为不可信输入，必须解析为严格类型并经过后端校验后才能展示为可执行 SQL。
- 前端只消费后端 API，不直接连接用户数据库。
