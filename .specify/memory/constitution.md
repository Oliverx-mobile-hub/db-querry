# db-querry 项目宪章

## 核心原则

### I. 只读数据库安全

所有生成的 SQL 和用户手写提交的 SQL 都必须被视为不可信输入。
系统默认只允许只读数据库操作。

后端必须拒绝破坏性或修改性语句，包括但不限于：
INSERT、UPDATE、DELETE、DROP、ALTER、TRUNCATE、CREATE、REPLACE、GRANT、REVOKE、EXEC 以及存储过程调用。

数据库凭据绝不能暴露给前端。查询执行只能通过后端完成。

### II. Go 后端负责业务逻辑

后端必须使用 Go 实现，并负责所有业务规则、SQL 校验、数据库访问、权限检查、查询执行和错误规范化。

前端代码不应重复实现后端校验规则，除非只是用于轻量级的用户体验提示。

后端模块应按职责拆分：
API 处理器、服务层、SQL 校验、数据库连接器、配置和日志。

### III. Vue 3 + Element Plus 前端作为薄客户端

前端必须使用 Vue 3 + Element Plus 实现。

前端负责用户交互、查询编辑、结果展示、加载状态和错误展示。
SQL 生成、SQL 校验和查询执行都必须调用后端 API 完成。

前端状态应清晰、可预测。避免把重要查询状态隐藏在隐式全局状态中。

### IV. API 契约优先

前后端集成必须基于清晰的 API 契约。

每个跨越前后端边界的功能都必须定义：
请求结构、响应结构、错误结构、加载行为和空结果行为。

所有后端返回给前端的 JSON 字段名必须使用 camelCase 格式。

破坏性 API 变更必须先体现在 spec 和 plan 中，然后再实现。

### V. 严格类型

前后端都必须使用严格的类型标注。

Go 后端必须使用明确的结构体、接口和错误类型表达 API 请求、API 响应、配置、数据库结果和业务状态。

Vue 3 前端必须使用 TypeScript。组件 props、emits、API 请求、API 响应、表格数据、表单状态和错误状态都必须有明确类型。

禁止在核心业务路径中使用无约束的动态类型。确实需要使用时，必须在代码中尽快收窄为明确类型。

### VI. 测试关键路径

关键后端行为必须有测试覆盖，尤其是：
SQL 只读校验、禁止语句检测、数据库执行边界、API 错误处理和权限检查。

在可行范围内，前端测试应覆盖核心用户流程：
提交自然语言、预览生成的 SQL、执行 SQL、查看结果和处理错误。

## 技术约束

- 后端语言：Go。
- 前端框架：Vue 3 + Element Plus。
- 前端语言：TypeScript。
- 前后端都必须有严格的类型标注。
- SQL 执行默认必须是只读的。
- 所有后端生成并返回给前端的 JSON 数据，字段名必须使用 camelCase 格式。
- 后端必须使用基于环境变量或环境配置的方式管理数据库凭据和服务密钥。
- 前端不得包含数据库凭据、模型密钥或高权限配置。
- 当前项目不需要 authentication，任何用户都可以使用。
- 不应为了用户身份、登录态、角色权限或会话管理引入认证系统。
- 因为没有认证，后端必须通过 SQL 安全校验、只读限制、查询超时、结果行数限制和错误脱敏来控制风险。
- 所有 API 响应应使用一致的 JSON 结构。
- 返回给用户的错误应易于理解，但不得泄露密钥、堆栈信息或原始连接详情。

## 开发流程

非简单功能必须先写 spec，再进入实现。

每个功能应按以下顺序推进：

1. 编写或更新功能 spec。
2. 澄清开放问题。
3. 编写实现计划。
4. 拆分开发任务。
5. 实现后端和前端变更。
6. 运行相关测试，或记录无法运行的原因。

实现必须与 spec 保持一致。如果实现需要改变行为，应先更新 spec。

## 治理规则

本宪章适用于本仓库中的所有 spec、plan、tasks 和实现工作。

任何削弱只读 SQL 安全、向前端暴露数据库凭据、绕过后端校验、破坏严格类型约束、或改变“无需认证即可使用”产品原则的功能，都必须获得明确批准，并记录原因。

复杂性必须在 plan 中说明理由。优先选择简单、可维护的 Go 和 Vue 3 代码，避免过早抽象。
